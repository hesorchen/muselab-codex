# Codex-native user input requests

- **Status:** Implemented
- **Scope:** Phase 2
- **Protocol baseline:** `codex-cli 0.144.1` app-server API
- **API maturity:** Experimental in this protocol baseline

## Goal

Let a Codex turn pause for structured user input without keeping the inherited
MCP compatibility tool or adding a second model-facing tool protocol. App-server owns
the tool call and its result; muselab-codex only adapts the server-initiated
request to the browser.

## Protocol contract

The version-matched schema generated by
`codex app-server generate-json-schema` defines this exchange:

| Direction | Method or payload |
|---|---|
| app-server to client | `item/tool/requestUserInput` |
| Browser event | SSE `ask_user_question` |
| Browser to FastAPI | `POST /api/chat/answer/{thread_id}/{request_id}` |
| Client to app-server | `{answers: {question_id: {answers: string[]}}}` |

Each request contains native `threadId`, `turnId`, `itemId`, and question ids.
The browser keys its local state and response by question id rather than by
display text, so duplicate or translated question text cannot corrupt answer
correlation. Choice labels, descriptions, headers, free-form questions, and
secret-input hints are preserved.

The existing browser event and HTTP route names remain as a compatibility
adapter. They no longer target `backend/ask_user_question.py` or a
muselab-owned MCP server in the default runtime.

## Experimental capability boundary

In `codex-cli 0.144.1`, both the request and its payload types are marked
experimental. The main app-server connection therefore declares
`initialize.params.capabilities.experimentalApi=true`. This is the only
experimental capability required by this slice. The isolated MCP inventory
process does not opt in.

The opt-in is explicit in this spec because the repository otherwise targets
stable app-server methods. When the pinned CLI changes, regenerate the schema,
rerun the captured protocol tests, and remove the opt-in if the method has
become stable.

## Lifecycle and fallback

`CodexClientRequestRouter` keeps command/file approvals and user-input
requests in separate brokers. App-server request handling already runs outside
the JSONL reader task, so a pending browser decision does not prevent other
responses or notifications from being correlated.

The user-input broker returns an empty `answers` object when:

- app-server supplies `autoResolutionMs` and that deadline expires;
- the browser publisher is unavailable;
- the application is shutting down; or
- the 30-minute safety deadline expires.

An empty answer delegates the next step to Codex. There is deliberately no
fallback MCP tool and no second runtime import. Late or duplicate browser
submissions receive `404` and cannot resolve another request.

## Validation and secret handling

- Question ids must be non-empty and unique.
- Answer keys must exactly match the pending native question ids.
- Scalar browser answers are converted to one-element native string arrays.
- Question, option, answer-count, and answer-length bounds protect the
  long-lived process from malformed payloads.
- Secret inputs use a password field, are cleared from browser state after a
  successful submission, and are never persisted by the backend.
- Raw prompts, answer values, request payloads, and credentials are not logged.

## Verification

Offline tests cover capability negotiation, JSONL request/response
correlation, approval and user-input routing, SSE replay, the authenticated
answer endpoint, native answer shape, automatic resolution, malformed ids,
secret-request cleanup, and process shutdown. The fake app-server blocks a
turn on `item/tool/requestUserInput`, then verifies the exact answer returned
by the browser bridge.
